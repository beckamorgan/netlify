Milestone Two

Primary Objectives:

  1. Practice Git workflow
  2. Choose or narrow down team project idea(s)
  3. Inception phase activities for 1 or 2 project ideas to explore topic

Overall Requirements:

  • Brainstorming mindmap
  • Lightweight vision statement
  • List of needs & features
  • Initial architecture diagram
  • Other "easy picking" parts of the inception template
  • At least 2 accepted pull requests (PR) for each team member (except maintainer)

Introduction

This week is all about choosing and exploring your team project topic. If you have too many good ideas then you'll explore them in order to narrow it down to just one. If you don't have any good ones then this is a second week of coming up with a good one.

We're beginning the first phase of a project, which we call inception. Remember, as always, everything goes in your repository!

Tasks:

  1. Gather with your team for a meeting to go through some inception phase activities for your chosen topic. If you have more than one topic that you're still considering, do these for all of them. We'll be working from the inception phase template that you can find in the Week 2 module on Canvas. Remember, that isn't a cookie cutter or step-by-step template, but rather a list of activities, deliverables or questions to ask during a reasonably agile project inception.
    1. Begin with a brainstorming session about your topic. Your goal is to explore its maximum possible scope, both breadth and depth, with input from all team members and stakeholders in order to outline what the project could possibly be about. This is completely free-form; go in whatever direction feels right. “Pie in the sky” ideas are fine here, as are mundane things. Just explore, and capture it in a Mindmap. I use Miro for this but there are other suitable tools out there. Since we're meeting remote at the moment you'll want to use a cloud-based online tool for this where everyone can use it simultaneously, which Miro does allow.
    2. Next up, try to capture the essence of what you explored in the mindmapping exercise and reduce it to a one paragraph vision statement. A vision statement is used to establish a shared vision about what the project is. It is a short description of your project that briefly answers the following questions:
      • What is it? Include 3 or so main features.
      • What is it's purpose?
      • Why are you proposing this topic?
      • Are there any competitors or existing applications that do this? If so, list them and explain why your topic is better or still worth doing.
    3. Now, let's try to establish a reasonable initial scope for the project. Write a list of needs and features for your application. What can a user do with it? What will it do for a user? What problems will it solve (i.e. a feature that solves a problem). What needs do your users have? Are there needs for the system? Next week we'll try to turn this into a list of “Epic” user stories and non-functional requirements.

    4. Create a diagram that shows your initial system/network architecture for this project. Normally you wouldn't know this yet because you'd wait until you knew more about your project before designing the architecture that will enable your solution to be built. But we've already fixed the initial architecture by deciding on the ASP.NET Core MVC / SQL Server / Azure stack. If you already have an external API for your project, add that in, otherwise just recognize this is an initial architecture that will be extended as you decide on features and solutions.

      And precisely because this is only an initial design, do not simply generate a .png file and drop it in your repo. This needs to be a living document that you can update anytime and by anyone in your team as decisions are made. I recommend using an open source vector drawing tool and placing the source code in your repository. SVG is a good choice. Or, use an online tool that can be accessed by anyone in your team and then include a link in a suitable place in your repo.

    5. Lastly, look over the inception template and see if there are any other activities you're ready to tackle this week. If so, have a go at it. The more you can do now the better. Otherwise you'll do them next week.

  2. Make sure you're using our Git Workflow for all the documents you create or edit. It can be a good idea to practice on a throw-away repository if you don't want to risk messing things up in your real ones; consider doing this while all team members are on a Zoom or Discord session so you can all learn from each other and practice at the same time. And make sure to practice merge conflicts!

    Each team member needs to have at least 2 accepted pull requests visible on the official repo. The official maintainer does not need to have any PR's though, as they'll be the one processing all these.
